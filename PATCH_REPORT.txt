- rawDir=3 (SW) → sprite=6 (смотрит NE) — юнит едет боком/назад
- rawDir=7 (NE) → sprite=2 (смотрит S)  — юнит едет боком

Что исправлено:

1. FE_SCOUT_DIR_MAP (runtime_flags.js):
   - Старый: [2, 0, 0, 6, 6, 4, 4, 2] (копия harvester)
   - Новый:  [7, 0, 1, 2, 3, 4, 5, 6]
   - Формула: DIR_MAP[rawDir] = (rawDir + 7) % 8
   - Компенсирует поворот спрайтов на +1 CW

2. unitDir8 forceDir (main.js):
   - Добавлен scout и light_tank в цепочку forceDir внутри unitDir8()
   - Раньше forceDir применялся только к harvester и builder
   - Теперь FE_SCOUT_FORCE_DIR и FE_LIGHT_TANK_FORCE_DIR работают корректно
     при dev-калибровке через горячие клавиши

- rawDir=1 (SE) → sprite=0 → faces (0+1)%8=1=SE ✓
- rawDir=2 (S)  → sprite=1 → faces (1+1)%8=2=S  ✓
- rawDir=3 (SW) → sprite=2 → faces (2+1)%8=3=SW ✓
- rawDir=4 (W)  → sprite=3 → faces (3+1)%8=4=W  ✓
- rawDir=5 (NW) → sprite=4 → faces (4+1)%8=5=NW ✓
- rawDir=6 (N)  → sprite=5 → faces (5+1)%8=6=N  ✓

Задача: BOT-SCOUT-02D4 — sweep timing completion tuning
Дата: 2026-05-11
Lane: Review
Ветка: glm/bot-scout-02d4-sweep-timing-tuning (от sandbox/main)

Проблема:
После BOT-SCOUT-02D3 (edge-aware sweep) scout достигает базы игрока и начинает
sweep, но возвращается с reason='sweep_timeout' вместо reason='sweep_done'.
Наблюдение: sweeping с gameTime ~144 до ~166 (~22 сек = текущий timeout),
не хватает времени на 3 точки с movement + observe.

Причина:
FE_SCOUT02D_SWEEP_TIMEOUT_SEC=22 слишком тесный для 3 sweep points:
3 × (movement ~4-5 sec + observe 2 sec) ≈ 18-21 sec, почти нет запаса.
Дополнительно: ARRIVE_DIST=2 требует подойти очень близко (лишние секунды),
OBSERVE_SEC=2 даёт 6 sec чистого observe на 3 точки.

Что изменено:

1. FE_SCOUT02D_SWEEP_TIMEOUT_SEC: 22 → 30
   - Запас 8 сек на movement между точками + unpredicted pathfinding
   - 3 точки × (4-5 сек move + 1.5 сек observe) ≈ 16.5-19.5 сек — укладывается с запасом

2. FE_SCOUT02D_SWEEP_OBSERVE_SEC: 2 → 1.5
   - 1.5 сек достаточно для scan области view=7
   - Экономия 0.5 сек × 3 точки = 1.5 сек суммарно

3. FE_SCOUT02D_SWEEP_ARRIVE_DIST: 2 → 3
   - Считать «прибыл» на 1 тайл раньше — меньше микроменёвров у точки
   - При view=7 наблюдение с расстояния 3 тайла от целевой клетки всё равно
     покрывает нужную область (7-3=4 тайла от позиции scout до края view)

Итого: budget = 30 сек, расход ~16.5-19.5 сек → запас ~10-13 сек.

Задача: BOT-INTEL-01 — persist scout intel snapshot
Дата: 2026-05-11
Lane: Review
Ветка: glm/bot-intel-01-persist-scout-intel (от sandbox/main)

Проблема:
Scout проходит полный lifecycle (outbound → observing → sweeping → returning → cooldown),

Цель:
Создать game.enemyIntel — persist-объект с таймстемпами, который обновляется при

Задача: BOT-ATTACK-11 — enemy tanks use scout intel target point as attack rally
Дата: 2026-05-11
Lane: Review
Ветка: glm/bot-attack-11-intel-rally-target (от sandbox/main)

Цель:
Когда enemy light_tank attack target selection не находит цель через 10F1 vision
(visible/known), использовать game.enemyIntel как источник target point.
Танки двигаются к intel-координатам через FE_PATCH_08BSilentMoveTo (coordinate move),
а не через setLightTankAttackApproachGeneric (реальный object target).
При прибытии в район intel-точки — попытка найти visible target через 10F1 и
переключиться на attack. Если цель не найдена — очистка rally flag, обычная bot logic.

Root cause:
Текущая attack-chain не поддерживает point-target (без real building/unit id):
- FE_PATCH_07BGetHostileLightTankTargetKind требует target.kind/owner/hp
- setLightTankAttackApproachGeneric требует target.id для attackApproachTargetId
- FE_PATCH_06BResolveApproachTarget ищет target по id среди game.buildings/units
- FE_ATTACK10IsWaveTargetAlive ищет targetId среди реальных объектов
- Synthetic point object невозможен без cascade-изменений

Решение:
Intel rally — coordinate-based move через уже существующую FE_PATCH_08BSilentMoveTo.
Танки получают _attack11IntelRally флаг с координатами и source.
Не ломает ни одну существующую attack-функцию.

Priority:
1. confirmed_hq — game.enemyIntel.playerHqCenterX/Y (scout видел HQ визуально)
2. estimated_hq — game.enemyIntel.playerHqEstimateCenterX/Y (scout target metadata)
3. legacy_fallback — findBaseBuilding('player') как раньше

Что сделано:

1. FE_ATTACK11ChooseIntelTarget() — helper после FE_INTEL01UpdateFromScout:
   - Читает game.enemyIntel
   - Priority 1: confirmed HQ (playerHqSeen + playerHqCenterX/Y)
   - Priority 2: estimated HQ (playerHqEstimateCenterX/Y)
   - Возвращает { targetX, targetY, targetSource, targetReason, intelFreshnessSec,
     playerHqSeen, playerHqEstimateAvailable } или null

2. FE_PATCH_08BPrepareAttack ATTACK-01 fallback:

   - Новый: if (!target) → FE_ATTACK11ChooseIntelTarget() →

     если assignedCount > 0 → return true (attack phase) →
     если assignedCount === 0 или intel нет → legacy findBaseBuilding fallback
   - return true / state.phase='attack' только если assignedCount > 0
   - Не назначает unit.command='attack' вручную (SilentMoveTo ставит manual_move)
   - Назначает только eligible tanks из waveSlice (не все enemy tanks)

3. Intel rally arrival detection в bot tick (state.phase === 'attack'):
   - Primary: dist(tank, rally) <= 3 → arrived
   - Cleanup: pathDone + dist <= 6 → arrived (path_done_far)
   - При arrival: FE_PATCH_10F1SelectAttackDecision → если direct target → attack command
   - Если target не найден → очистить rally flag, bot logic продолжит

4. Minimal guards:
   - ATTACK-04 overChase/attackInvalid: intel rally tanks пропускают regroup
   - ATTACK-04 pressure defense: intel rally tanks не отзываются для защиты базы
   - Regroup phase: очистка _attack11IntelRally флага

5. Telemetry game._botAttack11:
   - targetSource, targetX/Y, usesIntelPoint
   - assignedCount, rallyActiveCount
   - rallyArrivedCount, rallyConvertedCount
   - fallbackUsed, lastFailureReason
   - playerHqSeen, playerHqEstimateAvailable

6. ATTACK-01 telemetry расширена:
   - source: 'vision' | 'intel_rally' | 'hq_fallback'
