# Map Resource Placement Notes — 2026-05-19

Status: supplemental roadmap note.

This document captures the desired resource-placement feel from the older prototype reference.

Use together with:

- `docs/project/ROADMAP_CURRENT.md`
- `docs/project/ROADMAP_RTS_FEEL_NOTES_20260519.md`
- `docs/gameplay/MAP_GENERATION_SPEC.md`

## 1. Reference intent

The older prototype had a useful resource-placement pattern:

- resource fields were visually grouped near the player's base/start area;
- the starter resource cluster was easy to notice immediately;
- small/medium deposits were readable through amount labels/debug numbers;
- deposits were concentrated enough that the first harvesters had obvious work;
- resources felt like a field/pocket, not isolated random crystals.

The current Next mapgen should recover this gameplay feel without copying old code blindly.

## 2. Starter resource pocket

Target:

- each player start should have a dense starter resource pocket;
- the pocket should be placed near the player's start corner / HQ side;
- it can be pushed closer to the map corner so the start area feels intentional;
- resources should appear as a group/field, not sparse random singletons;
- the first harvesters should have several obvious nearby deposits.

Starter pocket composition target:

- many `mineral_small` nodes;
- several `mineral_medium` nodes;
- no `mineral_large` directly inside the immediate safe/core start zone unless explicitly approved;
- no `mineral_infinite` near the HQ.

Gameplay constraints:

- do not overlap HQ footprint;
- do not overlap builder/harvester spawn cells;
- do not block the first construction area;
- keep a safe path for current straight-line harvester movement until runtime pathfinding exists;
- do not place starter resources behind mountains/obstacles;
- do not rely on pure random scatter.

## 3. Corner-biased placement

For the current single-player start, resource pockets may be biased toward the start corner.

Design direction:

- choose a resource pocket center near the start/HQ but shifted toward the nearest map corner;
- create several nearby deposit groups around that pocket center;
- keep a clear space immediately around HQ and unit spawn;
- keep the pocket close enough for quick first harvest cycles.

This creates the older RTS feel where the base is near one corner/side and the first mineral field is visually nearby.

## 4. Deposit field shape

Resource field should not be a perfect square or uniform grid.

Target shape:

- loose cluster/arc/patch;
- several deposits close enough to read as one field;
- small gaps between deposits for readability and movement;
- no exact straight line unless generated by a debug/test seed;
- avoid placing all crystals on the same diagonal.

Recommended generator idea:

```text
start pocket center
→ place 2–4 subclusters
→ each subcluster places 3–8 deposits
→ more small than medium
→ reject overlaps and blocked path cases
```

## 5. Resource amount/debug labels

The old reference showed numeric values above resources.

Decision:

- production UI does not need permanent labels above deposits right now;
- dev/debug mode should eventually be able to show resource amount/remaining value above deposits;
- useful for balancing `small`, `medium`, `large`, and `infinite` fields.

Possible future debug overlay:

- resource type;
- remaining amount;
- infinite marker;
- footprint outline;
- blocked/approach cells.

This belongs to:

```text
DEV-SANDBOX-ARCH-01
```

or a small:

```text
RESOURCE-DEBUG-OVERLAY-01
```

## 6. Center / mineral infinite zone

The center should not contain only one isolated infinite deposit.

Target:

- central `mineral_infinite` is the main 3x3 strategic deposit;
- around it should be additional resource fields;
- center should feel like a rich contested zone;
- `medium` and `large` deposits should be more common near the center than near start;
- central resources must remain reachable and not fully enclosed by obstacles.

Suggested center composition:

- one `mineral_infinite` 3x3 deposit near exact center;
- several `mineral_large` nodes around it;
- several `mineral_medium` nodes around it;
- few/no `mineral_small` unless used as filler;
- obstacle clusters nearby are allowed, but must not lock access.

## 7. Relation to obstacles/edge biome

Starter resource pocket and map edge obstacles must not fight each other.

Rules:

- edge mountains/rocks should frame the map, not block the starter pocket;
- starter resource pocket should remain accessible from HQ;
- if start is near a corner, edge obstacles should form background/border behind or beside the resource pocket, not cut it off;
- volcanoes should remain rare landmarks, not common blockers.

## 8. Implementation placement in roadmap

This belongs to:

```text
MAPGEN-QA-ARCH-01
```

Likely stages:

- Stage A — starter resource pockets / corner-biased placement;
- Stage B — center resource field around 3x3 mineral_infinite;
- Stage C — edge obstacle biome integration;
- Stage D — debug overlay/resource amount labels if bundled safely, otherwise later under DEV-SANDBOX.

Do not mix this with `TERRITORY-TUNING-01`; territory speed/radius is a small separate tuning PR.

## 9. Manual QA acceptance

A generated standard map should be acceptable if:

- player immediately sees a nearby resource pocket;
- pocket is close to HQ/start corner;
- there are enough small resources for early harvesting;
- there are some medium resources nearby;
- harvesters do not need to cross obstacles to start working;
- the field reads visually as a coherent resource area;
- center has a clear large/infinite resource objective;
- center contains more resources than only one isolated infinite node;
- edge mountains frame the map without blocking first economy.
