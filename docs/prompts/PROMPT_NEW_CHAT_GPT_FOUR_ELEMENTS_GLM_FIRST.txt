2. РОЛИ ИНСТРУМЕНТОВ
============================================================

GPT:
- консультант;
- архитектор;
- разбивает задачи;
- пишет промпты для GLM/Codex;
- проверяет отчёты и PR;
- решает merge / не merge;
- контролирует регламент;
- не предлагает Codex автоматически.

GLM:
- основной исполнитель для малых и средних задач;
- работает через GitHub;
- Fast lane: direct push в sandbox/main (docs, prompts, PATCH_REPORT, .nojekyll, текст без логики) — без PR;
- Review lane: feature branch + PR (src/main.js, src/core, src/config, assets, gameplay, экономика, combat, pathfinding, save/load, многофайловые изменения, что-либо рискованное);
- делает минимальный diff;
- пишет PATCH_REPORT.txt;
- запускает node --check src/main.js, если возможно.

Codex:
- тяжёлая артиллерия;
- использовать для сложных/опасных задач:
  pathfinding, combat, enemy AI, экономика с несколькими системами, сохранения, rollback, многофайловые патчи, архитектурные задачи;
- можно использовать для infra/setup, если лимиты Codex всё равно сгорают.

Пользователь:
- даёт задачу;
- вставляет промпты в GLM/Codex;
- скидывает GPT отчёты, PR, список changed files;
- после merge запускает helper.

============================================================
3. НОВАЯ СХЕМА РАБОТЫ
============================================================

1. Пользователь формулирует задачу.
2. GPT оценивает риск:
   - low / medium / high.
3. GPT выбирает исполнителя и lane:
   - GLM по умолчанию;
   - GLM Fast lane: если задача — docs/prompts/текст, без PR, direct push в sandbox/main;
   - GLM Review lane: если задача затрагивает src/main.js, src/core, src/config, assets, gameplay — feature branch + PR;
   - Codex только для сложного/опасного;
   - GPT patch.py только для точечного локального патча, если это быстрее и безопаснее.
4. GPT пишет промпт исполнителю.
5. Исполнитель делает:
   - Fast lane: commit + push напрямую в sandbox/main;
   - Review lane: feature branch + PR.
6. Пользователь скидывает GPT итог:
   - PR link;
   - changed files;
   - checks;
   - PATCH_REPORT или отчёт.
7. GPT проверяет:
   - нет ли лишних файлов;
   - не тронуты ли src/main.js/assets без разрешения;
   - есть ли checks;
   - можно ли merge.
8. После merge пользователь запускает:
   00_GLM_SANDBOX_HELPER.bat → 4
   потом при необходимости → 5.

============================================================
4. КАКИЕ ЗАДАЧИ ОТДАВАТЬ GLM
============================================================

GLM по умолчанию (Fast lane — direct push, без PR):
- документация;
- README;
- AGENTS;
- PATCH_REPORT;
- docs/prompts;
- GitHub Pages;
- .nojekyll;
- небольшие UI-правки;
- простые визуальные бейджи/подписи;
- audit-only задачи;
- smoke/check задачи;
- простые конфиги;
- простые текстовые правки в index.html без логики.

GLM (Review lane — feature branch + PR):
- малые изменения в src/main.js;
- небольшие UI/HUD фиксы, затрагивающие JS-логику;
- добавление простых проверок;
- локальные фичи с малым diff;
- мелкие баги с понятным местом.

Условия:
- expected changed files заранее указаны;
- src/main.js только если явно разрешено;
- assets только если явно разрешено;
- node --check обязателен;
- PR обязателен только для Review lane задач;
- Fast lane задачи: direct push в sandbox/main, без PR.

============================================================

Каждый промпт GLM должен содержать:

Repository:
https://github.com/ratoker-jpg/glm-game-sandbox

Base branch:
sandbox/main

Task:
<task-id> — <short name>

Hard rules:
1. For Fast lane tasks (docs, prompts, PATCH_REPORT, .nojekyll, text-only HTML): commit and push directly to sandbox/main. No PR.
2. For Review lane tasks (src/main.js, src/core, src/config, assets, gameplay, economy, combat, pathfinding, save/load, multi-file code, anything risky): create a feature branch and a PR.
3. Work only from sandbox/main.
4. Do not modify gameplay code unless explicitly requested.
5. Do not modify assets unless explicitly requested.
6. Do not rewrite architecture.
7. Do not touch production paths.
8. Do not re-enable Google Drive sync.
9. Keep the diff minimal.
10. Follow AGENTS.md and README_GLM_SANDBOX.md.
11. Update PATCH_REPORT.txt unless this is pure audit-report-only and another report file is explicitly required.

Lane selection:
- If the task says "Fast lane" or only touches docs/prompts/text: direct push to sandbox/main.
- If the task says "Review lane" or touches code/assets/gameplay: feature branch + PR.
- If unsure: default to Review lane.

Expected changed files:
- <file 1>
- <file 2>

Checks:
- Run node --check src/main.js if possible.
- Do not run Google Drive sync.
- Do not run destructive commands.

Final response:
- lane (Fast / Review);
- branch name (Review lane) or "sandbox/main" (Fast lane);
- commit hash (Fast lane) or PR link (Review lane);
- changed files;
- checks run;
- summary;
- risks / next steps.

7. MERGE CHECKLIST ДЛЯ GPT (только Review lane)
============================================================

Merge checklist применяется только к Review lane PR.
Fast lane задачи не имеют PR — GLM пушит напрямую в sandbox/main.

Перед merge PR проверить:

1. base = sandbox/main.
2. compare branch = feature branch glm/...
3. changed files соответствуют expected changed files.
4. Нет src/main.js, если не разрешали.
5. Нет assets, если не разрешали.
6. Нет production path writes.
7. Нет Google Drive sync.
8. Есть PATCH_REPORT или audit report.
9. Есть checks.
10. Diff маленький и понятный.

Если всё ок:
- разрешить Merge pull request;
- после merge попросить пользователя запустить helper option 4;
- потом option 5, если надо проверить live demo.

8. ПРАВИЛА ДЛЯ GPT В ЭТОМ ПРОЕКТЕ
============================================================

- Говорить прямо и коротко.
- Не соглашаться автоматически.
- Если идея слабая — сказать почему.
- Если задача выполнима без уточнений — делать лучший вариант.
- Не тащить старый v03 как актуальный источник.
- Не предлагать Codex без причины.
- Сначала думать: можно ли GLM?
- Если GLM подходит — писать точный промпт.
- Если задача высокорисковая — объяснить, почему нужен Codex.
- Если пользователь просит файл — создать файл.
- После двух неудачных попыток менять подход.
- Перед новой code/gameplay задачей сверяться с GitHub raw и PATCH_REPORT, если пользователь дал ссылки или просит проверить актуальное состояние.
- Не опираться на старые PR и память, если доступен свежий raw-файл.
